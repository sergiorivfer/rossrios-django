Aquí va el resumen metódico, estructurado y sin saltos de lo que hemos hecho en el proyecto RossRios (migración WordPress → Python/Django). Lo dejo por fases + decisiones + problemas/soluciones, para que sirva como bitácora real.


0) Contexto y motivo del proyecto (por qué existe esto)
RossRios hoy funciona en WordPress, pero empezó a quedarse corto por:
Problemas con emails/CC (correos secundarios no siempre llegaban).
Backend lento (timeouts al editar/actualizar órdenes).
Quejas por límite bajo de subida de archivos.
Se decidió priorizar RossRios y obviar otros sitios.
Para no depender de WordPress y reducir futuros “incendios”, arrancamos un rebuild en Django como proyecto paralelo (extra), usando ~10% del tiempo para ir dejando lista una arquitectura más robusta.
1) Arquitectura que montamos (lo que corre hoy)
Infra (Docker)
Proyecto corre completamente en Docker Compose con 4 servicios:
web (Django)
worker (Celery)
db (Postgres)
redis (broker/result backend de Celery)
Se usa .env para configuración de:
Postgres
Redis
Email (Brevo SMTP)
listas de destinatarios admin
Estructura del repo
Apps Django:
orders (núcleo: pedidos, archivos, estados, staff dashboard)
accounts (signup/login/logout + región CA/US por session)
config (settings, urls, celery, wsgi/asgi)
Templates:
templates/accounts/signup.html
templates/registration/login.html
templates/orders/make_order.html
templates/orders/order_status.html (listado)
templates/orders/order_detail.html (detalle)
templates/orders/staff/* (dashboard, orders list, calendar, staff detail)
Media local:
media/order_files/...
media/order_pdfs/<public_id>/client.pdf y admin.pdf
2) Fase 0 — Infra y encendido (Docker + Django + DB + worker)


✅ Hecho


Levantamos servicios con Docker Compose.

Corregimos errores iniciales de settings:

3) Fase 1 — Modelo de datos + Admin base


✅ Hecho (y usable)


Modelo de datos (orders/models.py)


Creamos 3 modelos:


Order
Identificador público public_id (UUID).
Relación customer (usuario que crea la orden).
Región CA/US.
Datos cliente: nombre, teléfono, email primario + secundario + tercero + cuarto.
Datos order: order_name, tax_number, address, order_number.
Supplier pickup como lista (JSON) con validación (incluye opción None).
Rush/Standard, due_date.
Delivery location con opción Other + campo adicional obligatorio si aplica.
Notas (cliente e internas).
Status (processing, on_hold, in_production, completed, cancelled).
URLs de PDFs (client/admin).
OrderFile
Archivos asociados a una orden.
Guarda nombre original, tamaño, usuario que subió.
Validaciones:
extensiones permitidas
tamaño máximo por archivo (inicialmente 3MB)
OrderStatusHistory
Historial de cambios de estado: old_status → new_status + quién cambió + timestamp + nota.
Admin (orders/admin.py)
Personalizamos admin para que sea operable:
OrderAdmin con columnas, filtros, búsqueda.
Inline de archivos (OrderFileInline).
Form custom para supplier_pickup con checkboxes (CheckboxSelectMultiple) + validación “None no se mezcla”.
save_model registra OrderStatusHistory cuando cambia status.


Problemas que resolvimos aquí


AlreadyRegistered (modelo registrado dos veces) → se eliminó registro duplicado.
Errores de migración por campos non-nullable nuevos:
cuando agregaste delivery_location, Django pidió default por datos existentes → se manejó ajustando migraciones/valores.
Error “column does not exist” en admin por cambios en modelo sin migrar correctamente → se ordenó el flujo de makemigrations + migrate.

6) Fase 4 — PDF + Email automático (Celery + Brevo SMTP)


✅ Hecho en core (PDFs + emails cliente/admin + retry/log)


Celery
config/celery.py configurado con:
app.config_from_object(... namespace="CELERY")
app.autodiscover_tasks()
Se verificó worker con task ping (y se vieron logs de ejecución).
PDFs
Task generate_order_pdfs:
genera client.pdf y admin.pdf con render_order_pdf(order, kind=...)
guarda en media/order_pdfs/<public_id>/
actualiza order.client_pdf_url y order.admin_pdf_url
con autoretry (backoff, jitter, max_retries=5)
Confirmamos por consola que PDFs existían (ls /app/media/order_pdfs/...).
Emails
Se implementó envío para:
Cliente: email + PDF cliente (y soporta 1–4 correos)
Admin: email + PDF admin (destinatarios desde .env)
Se probó primero con console.EmailBackend (logs imprimían el MIME completo).
Luego se configuró Brevo SMTP real:
se agregó .env con credenciales SMTP
se corrigió que web y worker realmente cargaran variables (verificación con printenv y manage.py shell)
Se confirmó recepción real en Gmail (admin recibió).


Problemas que resolvimos aquí


Worker usaba console backend al inicio → settings/env no estaba aplicando como creías → se verificó dentro de contenedores.
Error SSL TLS (hostname mismatch) en una prueba → se ajustó configuración hasta que SMTP entregó emails reales.
Admin emails “skipped” cuando ADMIN_ORDER_EMAILS estaba vacío → se corrigió .env y settings para parsear lista.
Comportamiento raro con Gmail (a veces no llega a un correo específico) → se concluyó que era filtrado/entrega, no lógica (porque logs mostraban envío + otros destinatarios sí recibían).

8) Fase 6 — CA/US + multiidioma + producción (EN PROGRESO)


⏳ En progreso (estamos en la tarea 6.1)


Lo que empezamos (6.1 CA/US)
Objetivo: selector CA/US + persistencia (session/cookie) + “region” consistente al guardar órdenes.
Se añadió set_region (POST) en accounts.
Se empezó a forzar order.region = request.region para que no dependa del POST.


Errores que aparecieron y se resolvieron


transaction no importado → NameError en make-order → se importó from django.db import transaction.
chain no definido/import → NameError por encadenar tasks → se importó from celery import chain.
Caídas de web/worker por errores en accounts/urls/views (ya arreglados).
Error grave de Docker Desktop:
overlayfs metadata.db input/output error → era infra, no código.
Se resolvió reiniciando/limpiando el engine (WSL/Docker) y volvió a levantar.


Pendiente real de Fase 6


Middleware RegionMiddleware (o consolidarlo) para asegurar request.region siempre definido.
Selector visible en templates (partial topbar).
Decidir estrategia final:
por dominio/subdominio (ca./us.) vs selector en session
Multi-idioma (EN/FR/ES) aún no implementado.
Producción (S3/Spaces, settings prod, deploy, backups) aún no implementado.
